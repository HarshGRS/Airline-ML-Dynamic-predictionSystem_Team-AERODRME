# Complete Project Audit — Airline ML Dynamic Pricing System

This document explains every file currently in the repository, why it exists, how it connects to everything else, and what's wrong with it. Written for someone joining with zero context.

---

## 0. The one-paragraph version

This repo contains **two generations of the same idea**. Generation 1 (`backend/src/`, `backend/infrastructure/`, `backend/scripts/`, and 11 of the 12 React components) is a **non-functional AWS-architecture mockup**: real code, but it talks to AWS services that were never deployed, and the frontend renders `Math.random()` numbers. Generation 2 (`backend/app/`, `backend/tests/`, `backend/alembic/`, `frontend/src/api/`, `frontend/src/components/PricePrediction.js`) is a **working system**: a real XGBoost model trained on a real dataset, served by a real FastAPI backend with real auth and a real database, with one real frontend page wired to it. The two generations currently coexist. Generation 1 is dead weight that should eventually be deleted; it's documented below for completeness, not because it's load-bearing.

## 3. Data flow diagram (a single /predict request, end to end)

```
1. User fills PricePrediction.js form (airline, cities, times, stops, class, duration, days_left)
2. onSubmit → predictPrice() in frontend/src/api/client.js
3. axios POST http://localhost:8000/api/v1/predict  (body = form fields)
4. FastAPI routes to app/api/v1/predict.py::predict_price()
5. FastAPI validates body against schemas/predict.py::PredictRequest
   - enum fields rejected if not in the 6 cities / 6 airlines / etc.
   - cross-field validator rejects source_city == destination_city
6. Depends(get_prediction_service) returns the singleton PredictionService
   (loaded once at app startup: model.json + encoder.joblib + model_metadata.json)
7. PredictionService.predict():
   a. builds a 1-row DataFrame in FEATURE_COLUMNS order
   b. FeaturePipeline.transform() ordinal-encodes the categoricals
      (same encoder object used at training time — no train/serve skew)
   c. model.predict() → raw price float
   d. out-of-distribution check: z-score of duration/days_left vs
      training mean/std stored in model_metadata.json
   e. confidence interval = price ± MAE (±2×MAE if OOD)
8. Returns PredictResponse (predicted_price, confidence_low/high,
   model_version, out_of_distribution)
9. Frontend renders the price card + (separately) calls GET /model/info
   to show real feature importances and real held-out accuracy
```

## 4. Authentication flow

```
SIGNUP                                   LOGIN
------                                   -----
POST /auth/signup                        POST /auth/login
  {email, password}                        {email, password}
  ↓                                         ↓
check email not already in users table    look up user by email
  ↓                                         ↓
bcrypt.hashpw(password) → hashed_password  bcrypt.checkpw(password, stored hash)
  ↓                                         ↓
INSERT INTO users                          if mismatch → 401
  ↓                                         ↓
create_access_token(subject=email)         create_access_token(subject=email)
  → jwt.encode({sub: email, exp: +24h},      (same as signup)
     SECRET_KEY, HS256)
  ↓                                         ↓
return {access_token, token_type: bearer}  return {access_token, token_type: bearer}

EVERY SUBSEQUENT REQUEST TO A PROTECTED ROUTE
----------------------------------------------
Header: Authorization: Bearer <token>
  ↓
FastAPI Depends(get_current_user)
  ↓
oauth2_scheme extracts token from header (None if missing → 401 immediately)
  ↓
decode_access_token(): jwt.decode() with SECRET_KEY
  - raises 401 if signature invalid or token expired
  ↓
look up User by email (the JWT "sub" claim)
  - raises 401 if user no longer exists
  ↓
route handler receives `current_user: User` — e.g. used to filter
  `WHERE user_id = current_user.id` so users only ever see their own rows
```

No refresh tokens, no token revocation list, no password-reset flow — flagged in the weaknesses section.

## 5. ML pipeline flow

```
OFFLINE (run manually: python -m app.ml.train)
-----------------------------------------------
Clean_Dataset.csv (300,153 rows, Kaggle)
  ↓ load_raw_dataset() — drops the unnamed index column
  ↓ train_test_split(test_size=0.2, random_state=42)
  ↓ FeaturePipeline.fit_transform() on TRAIN only
       OrdinalEncoder fit on: airline, source_city, departure_time,
       stops, arrival_time, destination_city, class
  ↓ XGBRegressor(n_estimators=300, max_depth=6, lr=0.08).fit()
  ↓ evaluate on TEST split (never seen during fit/encode):
       MAE=2137, RMSE=3742, MAPE=15.06%, R²=0.9728
  ↓ persist:
       model.json           (XGBoost native format)
       encoder.joblib        (fitted OrdinalEncoder)
       model_metadata.json   (metrics, feature importances, numeric stats)
       docs/model_card.md    (human-readable report)

ONLINE (every FastAPI process startup)
---------------------------------------
PredictionService.__init__() runs ONCE per process:
  - XGBRegressor().load_model(model.json)
  - FeaturePipeline.load(encoder.joblib)
  - json.loads(model_metadata.json)
This is why /health also calls get_prediction_service() — a corrupted
artifact fails fast at startup/health-check, not on the first user request.
```

**Current (actual, as of this audit):** there is no deployment. Both servers run locally: `uvicorn app.main:app --port 8000` and `npm start` on :3000. `backend/Dockerfile` exists but has never been built — Docker Desktop's daemon wasn't running when it was authored, so it is **unverified code**, not a working artifact.

**Planned (per V2_BLUEPRINT.md, not yet executed):** push to GitHub → Vercel auto-deploys `frontend/` on push to `main` → Render/Fly.io builds `backend/Dockerfile` and deploys → both connect to a free-tier Postgres (Neon/Supabase) by setting `DATABASE_URL`, which is the only code change needed since all DB access goes through SQLAlchemy.

## 7. File-by-file audit — `backend/app/core/` (cross-cutting infrastructure)

This folder holds nothing domain-specific (no flights, no users-as-a-concept) — it's the plumbing every other module imports. That's the **separation-of-concerns** pattern: business logic in `models`/`schemas`/`services`/`api`, infrastructure concerns isolated here so they can be swapped without touching business code.

### `app/core/config.py`
1. **Why created:** centralize environment-dependent settings (app name, API prefix, CORS origins, model version) instead of scattering string literals across files.
2. **Problem solved:** without it, changing the CORS-allowed origin for production would mean grepping every file for `"http://localhost:3000"`.
3. **What breaks if removed:** `main.py` fails to import (`from app.core.config import settings`); the entire app fails to start.
4. **Connections:** imported by `main.py` for CORS middleware and the API prefix used by every router.
5. **Inputs:** environment variables (currently none are actually overridden — all defaults are hardcoded as Pydantic field defaults).
6. **Outputs:** a single `settings` singleton object.
7. **Dependencies:** `pydantic_settings.BaseSettings`.
8. **Depended on by:** `main.py`.
9. **Design pattern:** Settings/Configuration object (singleton via module-level instantiation).
10. **Alternatives:** plain `os.environ.get()` calls everywhere (worse — no validation, no single source of truth); `python-dotenv` + raw dict (less type-safe).
11. **Necessary or over-engineered:** necessary, and actually under-engineered for production — `cors_origins` should read from an env var so a deployed frontend origin can be added without a code change.
12. **Recruiter read:** barely visible, but its existence (rather than scattered literals) is a quiet "this person has shipped software before" signal.
13. **Senior engineer review:** would flag that `JWT_SECRET_KEY` lives in `core/security.py` via `os.getenv` instead of also being a field on this `Settings` object — inconsistent, two different config patterns in the same codebase.
14. **Improvement:** move `JWT_SECRET_KEY`, `DATABASE_URL` into `Settings` for one consistent config surface; make `cors_origins` env-overridable (`Field(default=...)` reading a comma-separated env var).

### `app/core/database.py`
1. **Why created:** single place that decides *what* database engine the whole app talks to and hands out sessions.
2. **Problem solved:** without a shared `engine`/`SessionLocal`, every file that touches the DB would open its own connection with its own URL — guaranteed inconsistency.
3. **What breaks if removed:** every model file (`Base` import), every API route (`get_db` dependency), and Alembic's `env.py` fail to import. The app cannot start.
4. **Connections:** `Base` is the declarative base every SQLAlchemy model inherits from; `get_db` is a FastAPI dependency injected into every route that touches the DB; `DATABASE_URL` is also read directly by `alembic/env.py`.
5. **Inputs:** `DATABASE_URL` env var (defaults to `sqlite:///./app.db`).
6. **Outputs:** an `engine`, a `SessionLocal` factory, a `get_db()` generator yielding a `Session`.
7. **Dependencies:** `sqlalchemy`.
8. **Depended on by:** `models/*.py`, `api/v1/auth.py`, `watchlists.py`, `saved_searches.py`, `core/security.py`, `alembic/env.py`.
9. **Design pattern:** Dependency Injection (the `get_db` generator pattern is FastAPI's idiomatic way to scope a DB session to one request and guarantee `.close()` via `finally`).
10. **Alternatives:** a global session (dangerous — shared mutable state across concurrent requests); an ORM-less raw-SQL layer (more control, much more boilerplate, no migrations story).
11. **Necessary or over-engineered:** necessary, not over-engineered — this is the minimum viable version of this file.
12. **Recruiter read:** seeing `connect_args = {"check_same_thread": False} if DATABASE_URL.startswith("sqlite")` signals the author understands SQLite's threading model, a small but real detail.
13. **Senior engineer review:** would ask why there's no connection pool sizing (`pool_size`, `max_overflow`) configured for the eventual Postgres target — fine for SQLite, will matter the day this points at Neon/Supabase's connection-limited free tier (flagged in V2_BLUEPRINT.md Part 14).
14. **Improvement:** add pool settings behind an `if not DATABASE_URL.startswith("sqlite")` branch before deploying to Postgres.

### `app/core/logging.py`
1. **Why created:** make backend logs machine-parseable (JSON) instead of Python's default unstructured text, so they could later be ingested by a real log aggregator (Sentry/Logtail, per the blueprint).
2. **Problem solved:** default `logging` output is hard to grep/filter/alert on in production; structured logs with consistent fields are a baseline observability practice.
3. **What breaks if removed:** `main.py`'s `configure_logging()` call fails to import; logging would fall back to Python defaults (app still runs, just with worse logs).
4. **Connections:** called once in `main.py` at import time; every module's `logging.getLogger(name)` then emits through this formatter.
5. **Inputs:** Python `LogRecord` objects from anywhere in the app.
6. **Outputs:** single-line JSON strings to stdout.
7. **Dependencies:** stdlib `logging`, `json`, `time`.
8. **Depended on by:** `main.py` (the only direct caller of `configure_logging()`).
9. **Design pattern:** custom `Formatter` subclass — standard `logging` extension point.
10. **Alternatives:** `structlog` or `python-json-logger` (mature libraries that do the same thing with more features — timestamps as ISO8601, context binding); hand-rolling it here is fine at this scale but is the kind of thing a senior engineer would ask "why not use the library" about.
11. **Necessary or over-engineered:** borderline over-engineered for a single-developer portfolio project — but it directly demonstrates an observability instinct, which is exactly the kind of detail call out on a resume.
12. **Recruiter read:** unlikely to be read directly, but its presence in the file tree (`core/logging.py`) is a positive signal during a skim.
13. **Senior engineer review:** would note timestamps are raw epoch-ms ints (`int(time.time()*1000)`) instead of ISO8601 — harder to read in raw log dumps, easy fix.
14. **Improvement:** swap to ISO8601 timestamps; add request-id correlation (e.g. via `contextvars`) so all log lines from one HTTP request can be grepped together.

### `app/core/rate_limit.py`
1. **Why created:** define the rate-limiter instance in one place so multiple route files can import and decorate with the same `limiter` without circular imports.
2. **Problem solved:** without a shared instance, each router would need its own limiter with its own in-memory counter — they wouldn't share rate-limit state for the same client.
3. **What breaks if removed:** `auth.py`, `watchlists.py`, `saved_searches.py`, and `main.py` (for the exception handler) all fail to import. App fails to start.
4. **Connections:** `main.py` registers it as `app.state.limiter`; route files import `limiter` to decorate handlers with `@limiter.limit("5/minute")`.
5. **Inputs:** the requesting client's IP (`get_remote_address`).
6. **Outputs:** raises `RateLimitExceeded` when a client exceeds the configured rate, caught by `main.py`'s exception handler and turned into a 429.
7. **Dependencies:** `slowapi`.
8. **Depended on by:** `app/api/v1/auth.py`, `watchlists.py`, `saved_searches.py`, `app/main.py`, `tests/test_rate_limit.py`, `tests/conftest.py` (disables it for the rest of the suite).
9. **Design pattern:** Token-bucket / fixed-window rate limiting (slowapi's default), Singleton (one shared `Limiter`).
10. **Alternatives:** Redis-backed rate limiting (necessary once there's more than one backend process — in-memory counters don't share state across processes/instances) — this is explicitly an in-memory, single-process limiter today.
11. **Necessary or over-engineered:** necessary for a backend exposed to the internet; correctly minimal for a single-instance deployment.
12. **Recruiter read:** "rate limiting" in a project README is a checkbox; seeing it actually tested (`test_rate_limit.py` trips a real 429) is the part that differentiates a real implementation from a claimed one.
13. **Senior engineer review:** would immediately flag that this breaks the moment the app scales to 2+ instances behind a load balancer (each instance has its own counter, so the *effective* limit becomes limit×instance-count) — correctly called out in V2_BLUEPRINT.md Part 14 as a known bottleneck.
14. **Improvement:** swap `Limiter(key_func=get_remote_address)` for a Redis storage backend (`storage_uri="redis://..."`) the moment there's more than one backend process — Upstash's free tier makes this a $0 change.

### `app/core/security.py`
1. **Why created:** implement authentication primitives — password hashing, JWT issuance/verification, and the `get_current_user`/`require_admin` dependencies every protected route needs.
2. **Problem solved:** turns "users can sign up and log in" from a UI mockup into an actual access-control mechanism with cryptographic guarantees (bcrypt for passwords, HMAC-signed JWTs for sessions).
3. **What breaks if removed:** every protected endpoint (`/auth/me`, all of `/watchlists`, all of `/saved-searches`) fails to import — the entire authorization layer disappears.
4. **Connections:** `hash_password`/`verify_password` used by `api/v1/auth.py`; `create_access_token` used by signup/login; `get_current_user` injected as a `Depends()` into every protected route; `require_admin` is defined but **not currently used anywhere** (dead code — there is no admin-only endpoint yet, despite the V2_BLUEPRINT.md design calling for one).
5. **Inputs:** plaintext passwords (hashing), JWT bearer tokens (verification).
6. **Outputs:** bcrypt hash strings, signed JWT strings, decoded `User` ORM objects, or `HTTPException(401/403)`.
7. **Dependencies:** `bcrypt`, `python-jose`, `fastapi.security.OAuth2PasswordBearer`, `app.core.database`, `app.models.user.User`.
8. **Depended on by:** `api/v1/auth.py`, `watchlists.py`, `saved_searches.py`.
9. **Design pattern:** Dependency Injection (`Depends(get_current_user)`), Strategy (swappable hashing/token backend in principle, hardcoded in practice).
10. **Alternatives:** Supabase Auth or Firebase Auth (outsource this entire file to a vendor SDK — less code, less resume value, an external dependency and account). Rolling it by hand, as done here, is more work but a stronger interview talking point.
11. **Necessary or over-engineered:** necessary; not over-engineered — it's actually missing a few standard pieces (see improvements).
12. **Recruiter read:** this is one of the highest-value files in the repo to walk through in an interview — it proves understanding of password hashing, stateless auth, and dependency-based authorization, not just "I called an SDK."
13. **Senior engineer review:** would flag three real issues: (a) `SECRET_KEY` has an insecure hardcoded dev default that would be catastrophic if accidentally deployed without overriding `JWT_SECRET_KEY` — should fail loudly in a "production" mode rather than silently using the fallback; (b) no refresh-token mechanism, so the only way to extend a session past 24h is to log in again; (c) `require_admin` is unused dead code — either wire it to a real admin endpoint or delete it.
14. **Improvements:** add a startup check that raises if `ENV=production` and `JWT_SECRET_KEY` is unset; add token refresh; either delete `require_admin` or build the admin usage-analytics endpoint from the blueprint that would actually use it.

## 8. File-by-file audit — `backend/app/models/` (database tables)

**Folder purpose:** defines the persistent schema — what tables exist, their columns, and their relationships. Every row that survives a restart lives because of one of these three files.

### `app/models/user.py` → table `users`
1. **Why created:** every other feature (watchlists, saved searches) needs an owner; `users` is the root of that ownership graph.
2. **Problem solved:** without persisted identity, "save this search" is meaningless — there's nothing to save it *to*.
3. **What breaks if removed:** `Watchlist`/`SavedSearch` foreign keys (`ForeignKey("users.id")`) fail at table-creation time; `core/security.py`'s `get_current_user` fails to import.
4. **Connections:** `watchlists.user_id` and `saved_searches.user_id` both point here; `core/security.py` queries this table by email on every authenticated request.
5. **Inputs:** `email`, `hashed_password` at creation (from `api/v1/auth.py::signup`).
6. **Outputs:** rows queried by email (login) or by JWT subject (every protected request).
7. **Dependencies:** `sqlalchemy`, `app.core.database.Base`.
8. **Depended on by:** `app/models/__init__.py`, `app/core/security.py`, `app/api/v1/auth.py`, `alembic/env.py` (for autogenerate to see this table), both `watchlist.py`/`saved_search.py` (FK target).
9. **Design pattern:** Active Record–adjacent (SQLAlchemy declarative model), not a pure Active Record since business logic lives in services/routes, not on the model itself.
10. **Alternatives:** outsourcing identity entirely to Supabase Auth (no local `users` table at all, just a JWT issued by Supabase) — would mean no FK relationship to local data, complicating `watchlists`/`saved_searches` ownership.
11. **Necessary or over-engineered:** necessary; intentionally minimal (no `username`, `full_name`, `avatar_url` — none of which any current feature needs yet).
12. **Recruiter read:** unremarkable in isolation — it's the foundation everything else stands on, not a flashy file.
13. **Senior engineer review:** would note there's no `updated_at`, no soft-delete (`is_active`), and no unique constraint beyond `email` (fine for now, would matter for account recovery flows later).
14. **Improvement:** add `is_active: bool` before building any account-deletion/ban feature — deleting a `User` row currently cascades and hard-deletes their watchlists/saved searches (`cascade="all, delete-orphan"`), which is destructive and unrecoverable.

### `app/models/watchlist.py` → table `watchlists`
1. **Why created:** persist "alert me if Delhi→Mumbai drops below ₹X" — the literal feature from the V2_BLUEPRINT.md feature list.
2. **Problem solved:** turns "Watchlists" from a sidebar label with no backing data into a real, queryable, per-user record.
3. **What breaks if removed:** `api/v1/watchlists.py` fails to import entirely; the watchlist feature disappears.
4. **Connections:** FK to `users.id`; conceptually feeds the not-yet-built Phase 5 price-alert job (`price_history` + a scheduled comparison against `target_price` — that job doesn't exist yet, this table is ready for it).
5. **Inputs:** `source_city`, `destination_city`, `target_price` from `schemas/watchlist.py::WatchlistCreate`.
6. **Outputs:** rows filtered by `user_id` on list/delete.
7. **Dependencies:** `sqlalchemy`, `app.core.database.Base`.
8. **Depended on by:** `app/models/__init__.py`, `app/api/v1/watchlists.py`, the Alembic migration.
9. **Design pattern:** the docstring itself documents a deliberate simplification — storing `source_city`/`destination_city` as plain strings instead of a `route_id` FK into a `routes` table (which V2_BLUEPRINT.md's schema design calls for). This is a conscious scope cut, not an oversight.
10. **Alternatives:** the full normalized version with a `routes` table (better once there's a real `price_history` table to join against for the alert job; pure overhead today since nothing else queries by route yet).
11. **Necessary or over-engineered:** necessary; correctly *under*-engineered relative to the blueprint's full schema — the right call for the feature's current scope.
12. **Recruiter read:** the in-code comment explaining *why* it deviates from the "textbook" schema is itself a positive signal — it shows the author can justify a simplification rather than cargo-culting a normalized schema nobody needs yet.
13. **Senior engineer review:** would ask "what enforces that `target_price` is sane relative to real prices for that route?" — currently only bounded to `0 < price <= 200,000` at the Pydantic layer, with no relationship to actual observed prices (because there's no `price_history` table yet to check against).
14. **Improvement:** once Phase 5's price-polling job exists, add the scheduled comparison job that actually fulfills this table's purpose — right now it's a CRUD store with no consumer.

### `app/models/saved_search.py` → table `saved_searches`
1. **Why created:** let a user save a route+class combination they search often, to skip re-filling the form.
2. **Problem solved:** removes repetitive form-filling for repeat searches — same value proposition as Skyscanner/Google Flights "recent searches."
3. **What breaks if removed:** `api/v1/saved_searches.py` fails to import; feature disappears.
4. **Connections:** FK to `users.id`; otherwise standalone — nothing else currently reads from this table (no "re-run this search" button wired up in the frontend yet).
5. **Inputs:** `source_city`, `destination_city`, `flight_class` from `schemas/saved_search.py::SavedSearchCreate`.
6. **Outputs:** rows filtered by `user_id`.
7. **Dependencies:** `sqlalchemy`, `app.core.database.Base`.
8. **Depended on by:** `app/models/__init__.py`, `app/api/v1/saved_searches.py`, the Alembic migration.
9. **Design pattern:** same intentional simplification as `Watchlist` (no normalized routes table).
10. **Alternatives:** could have been merged into `Watchlist` as one generic "saved query" table with a nullable `target_price` distinguishing the two use cases — a real simplification opportunity (see refactoring section).
11. **Necessary or over-engineered:** the model itself is minimal and correct; having it as a *separate* table from `Watchlist` is arguably the over-engineered choice given how similar the two are.
12. **Recruiter read:** neutral.
13. **Senior engineer review:** would likely suggest merging `Watchlist` and `SavedSearch` into one table with an optional `target_price` column and a `kind` enum (`"alert"` vs `"saved"`) — same data, half the boilerplate, half the migration surface, half the test surface.
14. **Improvement:** the merge described above, deferred for now since the duplication is small (two tables, not ten) and the cost of merging mid-build outweighs the benefit today.

### `app/models/__init__.py`
1. **Why created:** so `from app.models import User, Watchlist, SavedSearch` works as one import, and critically, so **Alembic's autogenerate sees all three models** when `alembic/env.py` does `from app.models import SavedSearch, User, Watchlist` — SQLAlchemy's `Base.metadata` only knows about classes that have actually been imported somewhere.
2. **Problem solved:** a model file that's never imported is invisible to `Base.metadata.create_all()`/Alembic autogenerate even though the class exists on disk — this file is what makes all three tables "discoverable."
3. **What breaks if removed:** Alembic autogenerate would silently stop detecting two of the three tables; existing migrations would still apply, but any *future* schema change to `Watchlist`/`SavedSearch` wouldn't be picked up by `alembic revision --autogenerate`.
4. **Connections:** imported by `alembic/env.py`; transitively makes all models available via `app.models`.
5/6. **Inputs/Outputs:** none — pure re-export.
7. **Dependencies:** the three model files.
8. **Depended on by:** `alembic/env.py`, anything doing `from app.models import X`.
9. **Design pattern:** the "models package registry" pattern, standard in every SQLAlchemy + Alembic project.
10. **Alternatives:** explicitly importing each model file directly in `alembic/env.py` instead of through `__init__.py` — works identically, marginally less conventional.
11. **Necessary or over-engineered:** necessary, minimal.
12-14: not independently visible to a recruiter; a senior engineer would only notice it if it were *missing* (autogenerate silently breaking is a classic gotcha this file prevents).

## 9. File-by-file audit — `backend/app/schemas/` (request/response contracts)

**Folder purpose:** Pydantic models defining exactly what shape of JSON the API accepts and returns — the contract layer between the wire format and the Python/ORM objects in `models/`. This is what makes `/docs` (Swagger UI) auto-generate correctly, and what rejects malformed input before any business logic runs.

### `app/schemas/predict.py`
1. **Why created:** define the exact, finite set of valid inputs to `/predict` — six enums (`Airline`, `City`, `TimeOfDay`, `Stops`, `FlightClass`) instead of free-text strings, because the model was trained on exactly those categories and nothing else.
2. **Problem solved:** without strict enums, a user could submit `"airline": "DELTA"` (a real airline, just not one in the training data) and get a nonsense prediction silently, or worse, an encoder error. Enums make invalid input a 422 at the API boundary, not a runtime surprise three layers down in `PredictionService`.
3. **What breaks if removed:** `api/v1/predict.py`, `api/v1/model.py` (no, model.py doesn't use it), `services/prediction.py`, `schemas/watchlist.py` and `schemas/saved_search.py` (both import `City`/`FlightClass` from here) all fail to import.
4. **Connections:** the `City` and `FlightClass` enums defined here are **reused** by `schemas/watchlist.py` and `schemas/saved_search.py` — one source of truth for "what cities/classes exist in this system," not three copies that could drift out of sync.
5. **Inputs:** raw JSON body of a `/predict` POST.
6. **Outputs:** a validated `PredictRequest` object (input side) or a `PredictResponse` shape (output side, used as `response_model=` in the route).
7. **Dependencies:** `pydantic`.
8. **Depended on by:** `api/v1/predict.py`, `services/prediction.py`, `schemas/watchlist.py`, `schemas/saved_search.py`, `tests/conftest.py` (the `valid_predict_payload` fixture mirrors this schema), `tests/test_predict.py`.
9. **Design pattern:** Data Transfer Object / Value Object (Pydantic `BaseModel`), cross-field validation via `@field_validator`.
10. **Alternatives:** plain `dict` + manual `if` validation (far more error-prone, no auto-generated OpenAPI docs); dataclasses + `cerberus`/`marshmallow` (more boilerplate, less FastAPI-native).
11. **Necessary or over-engineered:** necessary; appropriately strict given the model's hard categorical limits.
12. **Recruiter read:** the enums matching the *actual training data's unique values* (verified against the CSV, not guessed) is the kind of detail that separates a real ML-serving API from a copy-pasted FastAPI tutorial.
13. **Senior engineer review:** would note the field alias trickery (`flight_class: FlightClass = Field(alias="class")`, since `class` is a reserved Python keyword) is handled correctly with `populate_by_name=True` — a subtle Pydantic v2 detail many get wrong.
14. **Improvement:** none urgent; could add a `model_config = {"json_schema_extra": {"example": {...}}}` to make the auto-generated `/docs` page show a ready-to-click example payload.

### `app/schemas/auth.py`
1. **Why created:** define what a signup/login request/response looks like, separate from the `User` ORM model — critically, `UserResponse` excludes `hashed_password`, so the password hash can never accidentally leak in an API response.
2. **Problem solved:** if routes returned the SQLAlchemy `User` object directly serialized, a careless `response_model` omission could leak password hashes; a separate schema makes that structurally impossible.
3. **What breaks if removed:** `api/v1/auth.py` fails to import entirely — no signup, no login, no `/me`.
4. **Connections:** `UserResponse.model_config = {"from_attributes": True}` lets Pydantic read directly off the SQLAlchemy `User` object's attributes — the bridge between ORM and API response.
5. **Inputs:** signup/login JSON bodies.
6. **Outputs:** `TokenResponse` (JWT), `UserResponse` (safe user projection).
7. **Dependencies:** `pydantic`, `pydantic[email]` (`EmailStr` requires the `email-validator` package, installed separately — a real dependency that's easy to forget).
8. **Depended on by:** `api/v1/auth.py`.
9. **Design pattern:** DTO, with an explicit security boundary (never expose `hashed_password`).
10. **Alternatives:** `exclude` lists on serialization (more error-prone — one missed `exclude` and the hash leaks; the separate-schema approach used here is strictly safer).
11. **Necessary or over-engineered:** necessary, correctly scoped.
12. **Recruiter read:** the password-length constraint (`Field(min_length=8, max_length=128)`) plus the separate-from-ORM response schema both signal security awareness without anyone needing to ask "did you think about this?"
13. **Senior engineer review:** would approve of the `from_attributes` bridge; might ask about password complexity rules beyond length (not enforced — flagged in weaknesses).
14. **Improvement:** add a password-strength check (not just length) if this were going further than a portfolio project.

### `app/schemas/watchlist.py` / `app/schemas/saved_search.py`
Both follow an identical shape and are documented together since the analysis is the same for both:
1. **Why created:** validate watchlist/saved-search creation payloads, reusing the `City`/`FlightClass` enums from `predict.py` so a watchlist can never reference a city the model doesn't know about.
2. **Problem solved:** prevents a watchlist being created for `"NewYork"` (not a real option) that would later be unactionable by any feature that reads this data.
3. **What breaks if removed:** `api/v1/watchlists.py`/`saved_searches.py` fail to import.
4. **Connections:** imports `City`/`FlightClass` from `schemas/predict.py` — these three schema files are coupled by design (single source of truth for valid cities).
5. **Inputs:** creation JSON bodies.
6. **Outputs:** `WatchlistResponse`/`SavedSearchResponse`, both `from_attributes=True` bridges onto the ORM models.
7. **Dependencies:** `pydantic`, `app.schemas.predict`.
8. **Depended on by:** their respective `api/v1/*.py` route files.
9. **Design pattern:** DTO with cross-field validation (`destination_city != source_city`), same pattern as `PredictRequest`.
10. **Alternatives:** as noted in the models section, these two schemas (and their backing tables) are similar enough to warrant merging.
11. **Necessary or over-engineered:** functionally necessary; structurally duplicated with each other (see refactoring opportunities).
12-14: see the `watchlist.py`/`saved_search.py` model entries above — the same "should these be merged" critique applies at the schema layer too.

## 10. File-by-file audit — `backend/app/api/v1/` (HTTP routes)

**Folder purpose:** the only layer that knows about HTTP — request parsing, status codes, dependency wiring. Business logic is deliberately kept thin here and pushed into `services/` (for prediction) or directly into simple ORM queries (for CRUD, since there isn't enough logic yet to warrant a service layer for watchlists/saved-searches).

### `app/api/v1/health.py`
1. **Why created:** give load balancers/orchestrators (and humans) one URL to hit to confirm the process is alive *and* the model loaded correctly.
2. **Problem solved:** without it, a corrupted model artifact would only surface as a 500 on the first real user's `/predict` call — this turns that into an immediate, observable startup failure.
3. **What breaks if removed:** no functional feature breaks, but the "fail fast on bad deploy" safety net (V2_BLUEPRINT.md Part 14) disappears, and any future uptime monitor/keep-warm cron has nothing to ping.
4. **Connections:** depends on `get_prediction_service` — the same singleton used by `/predict`.
5. **Inputs:** none (GET, no body).
6. **Outputs:** `{status, model_version, model_trained_at}`.
7. **Dependencies:** `app.services.prediction`.
8. **Depended on by:** `tests/test_health.py`; conceptually, any future deployment platform's health check config.
9. **Design pattern:** Health Check endpoint (standard cloud-native pattern).
10. **Alternatives:** a "dumb" health check that just returns `{"status": "ok"}` without touching the model — strictly worse, since it wouldn't catch a corrupted artifact.
11. **Necessary or over-engineered:** necessary, correctly minimal.
12. **Recruiter read:** low visibility but expected — its *absence* would be noticed by anyone checking for production-readiness.
13. **Senior engineer review:** approves of exercising the model rather than a no-op check; might ask for a DB ping too (currently doesn't touch the database at all).
14. **Improvement:** add a lightweight `SELECT 1` against the DB so this one endpoint verifies both dependencies (model + database) are healthy.

### `app/api/v1/predict.py`
1. **Why created:** the HTTP entry point for the core feature of the whole project — turning a flight description into a price.
2. **Problem solved:** exposes `PredictionService` (pure Python) over HTTP so the React frontend can call it.
3. **What breaks if removed:** the Price Prediction page in the frontend has nothing to call; the entire ML-serving story collapses.
4. **Connections:** thin wrapper — all real logic is in `services/prediction.py`; this file's only job is request/response wiring.
5. **Inputs:** `PredictRequest` (validated by FastAPI before this function body runs).
6. **Outputs:** `PredictResponse`.
7. **Dependencies:** `app.schemas.predict`, `app.services.prediction`.
8. **Depended on by:** `frontend/src/api/client.js::predictPrice()`, `tests/test_predict.py`.
9. **Design pattern:** thin Controller (in MVC terms) — no business logic, pure delegation.
10. **Alternatives:** inlining the prediction logic directly in this function — would work, but mixes HTTP concerns with ML concerns and makes the logic untestable without spinning up FastAPI (the actual design lets `PredictionService` be unit-tested directly, though no such direct unit test exists yet — only integration tests through the HTTP layer).
11. **Necessary or over-engineered:** correctly minimal — exactly as much code as needed and no more.
12. **Recruiter read:** the cleanliness of this file (7 lines of actual logic) next to the depth of `services/prediction.py` shows separation of concerns done right.
13. **Senior engineer review:** no notes — this is what a controller should look like.
14. **Improvement:** none for the file itself; the missing piece is a direct unit test of `PredictionService` that doesn't go through HTTP (faster, more isolated).

### `app/api/v1/model.py`
1. **Why created:** expose the trained model's real metrics/feature-importances over HTTP so the frontend's "Price Factors" panel shows actual numbers instead of invented ones.
2. **Problem solved:** without this, the frontend would have to either hardcode metrics (which go stale the moment the model is retrained) or have no "why this price" explanation at all.
3. **What breaks if removed:** `PricePrediction.js`'s "Top Price Drivers" and "Model Accuracy" panels have nothing to fetch — they'd need to fall back to hardcoded (i.e., fake) numbers, exactly the anti-pattern this whole rebuild is trying to eliminate.
4. **Connections:** reads the same `PredictionService.metadata` dict that `/predict` and `/health` use — all three endpoints share one loaded-once model object.
5. **Inputs:** none.
6. **Outputs:** `{model_version, trained_at, dataset, metrics, feature_importance}`.
7. **Dependencies:** `app.services.prediction`.
8. **Depended on by:** `frontend/src/api/client.js::getModelInfo()`.
9. **Design pattern:** thin Controller, same as `predict.py`.
10. **Alternatives:** baking this into the `/predict` response itself (would mean re-sending the same static metadata on every single prediction request — wasteful; a separate endpoint, fetched once per page load, is the right call).
11. **Necessary or over-engineered:** a genuinely good design choice, not over-engineered.
12. **Recruiter read:** this endpoint is *why* the demo can say "here's real SHAP-adjacent feature importance" instead of "trust me, it's a good model" — directly supports a resume claim.
13. **Senior engineer review:** no major notes; might point out `feature_importance` is raw XGBoost gain-based importance, not true SHAP values, despite `shap` being listed in `requirements.txt` and never actually used anywhere in the codebase yet.
14. **Improvement:** this is exactly the gap flagged in V2_BLUEPRINT.md's Feature #11 (SHAP explainability) — `shap` is installed but unused; wiring it in would upgrade this from "feature importance" (global, training-time) to "why *this* prediction" (local, per-request) explanations.

### `app/api/v1/auth.py`
1. **Why created:** the HTTP surface for signup/login/me — see the full auth-flow diagram in section 4.
2. **Problem solved:** turns `core/security.py`'s primitives into actual usable endpoints.
3. **What breaks if removed:** no user can ever sign up or log in; `watchlists.py`/`saved_searches.py` would be permanently inaccessible (their `get_current_user` dependency would have no way to ever have issued a valid token).
4. **Connections:** uses `core/security.py` for hashing/tokens, `core/database.py` for the session, `core/rate_limit.py` for throttling, `models.user.User` for the table, `schemas.auth` for request/response shapes.

8. **Depended on by:** the frontend (not yet wired — see weaknesses), `tests/test_auth.py`, `tests/conftest.py::auth_headers` fixture (used by every other test file needing an authenticated client).

10. **Alternatives:** Supabase/Firebase Auth SDK (less code, vendor lock-in, no local password storage to secure).
11. **Necessary or over-engineered:** necessary, well-scoped.
12. **Recruiter read:** the rate-limit decorators (`@limiter.limit("5/minute")` on signup) sitting right next to the route definition make the security posture visible at a glance during a code walkthrough.
13. **Senior engineer review:** would ask about email verification (none — anyone can sign up with `attacker@victim.com` and there's no confirmation step) and password reset (also none).
14. **Improvement:** email verification and password reset are the two biggest gaps for anything beyond a portfolio demo; both need an email-sending capability (Resend free tier, per the blueprint) that hasn't been wired in yet.

### `app/api/v1/watchlists.py` / `app/api/v1/saved_searches.py`
Documented together — structurally identical (create/list/delete, ownership-scoped, per-user cap).
1. **Why created:** the HTTP surface for the two persistence features described in the models section.
2. **Problem solved:** turns the DB tables into something the frontend can actually call.
3. **What breaks if removed:** the respective feature disappears entirely (404 on every route).
4. **Connections:** `get_current_user` for ownership, the respective model for queries, the respective schema for validation.

6. **Outputs:** created/listed/deleted resource, or 404 (not found *or* not owned by this user — deliberately the same response for both, to avoid leaking which IDs exist for other users), or 400 (cap reached), or 429 (rate-limited).

11. **Necessary or over-engineered:** functionally necessary; the duplication between the two files is the one clear "could be DRY-er" spot in the whole backend.
12. **Recruiter read:** the deliberate 404-not-403 choice on cross-user delete attempts (verified by `test_cannot_delete_another_users_watchlist`) is a subtle, correct security decision that a careful reviewer would notice and respect.
13. **Senior engineer review:** would flag the file-level duplication immediately and suggest either the model merge (section 8) or a shared generic CRUD helper.

## 11. File-by-file audit — `backend/app/services/prediction.py`

1. **Why created:** isolate all the *non-HTTP* logic of serving a prediction — loading artifacts, encoding features, computing confidence intervals, detecting out-of-distribution input — so it's testable independently of FastAPI and reusable from anywhere (a future batch-scoring script, for instance).
2. **Problem solved:** without this layer, `api/v1/predict.py` would have to load the model on every single request (catastrophically slow) or use some ad-hoc global-variable caching scattered through route code.
3. **What breaks if removed:** `predict.py`, `model.py`, and `health.py` (all three depend on `get_prediction_service`) fail to import — the entire serving path collapses.
4. **Connections:** the central hub of the ML-serving story — bridges `app/ml/features.py` (encoding), the persisted model artifacts, and the API layer.
5. **Inputs:** a `PredictRequest` object.
6. **Outputs:** a `PredictResponse` object.
7. **Dependencies:** `pandas`, `xgboost.XGBRegressor`, `app.ml.features`, `app.schemas.predict`.
8. **Depended on by:** `api/v1/predict.py`, `api/v1/model.py`, `api/v1/health.py`, `tests/conftest.py` (indirectly, since the test app loads it at import time).
9. **Design pattern:** Singleton (`get_prediction_service()` module-level cache via `_service: PredictionService | None = None` — the classic lazy-singleton pattern, here used specifically so the multi-megabyte model only loads once per process, not once per request).
10. **Alternatives:** FastAPI's `lifespan` context manager to load the model at startup explicitly into `app.state` (slightly more idiomatic for FastAPI specifically; the global-singleton approach used here works identically and is more portable outside FastAPI).
11. **Necessary or over-engineered:** necessary, and the out-of-distribution guard is a genuine "above and beyond a tutorial project" detail rather than over-engineering — it directly addresses a named risk from V2_BLUEPRINT.md Part 14.
12. **Recruiter read:** this file, walked through line by line, is the single best 5-minute interview demo in the repo — model loading, feature encoding reuse (train/serve parity), and a real OOD guard, all in ~75 lines.
13. **Senior engineer review:** would approve of the train/serve encoding reuse; might ask why the OOD check only covers the two numeric features (`duration`, `days_left`) and not unusual *categorical combinations* (e.g. a route/airline pair that never appeared together in training) — currently undetected since every categorical value individually is always "valid" by definition (it's in the enum), even if that specific combination is rare or absent in training data.
14. **Improvement:** the categorical-combination gap above; also, the module-level singleton makes this code harder to test with a *different* model in isolation (tests currently all share the one real trained model — acceptable here since the model is deterministic and fast to load, but worth knowing as a limitation).

## 12. File-by-file audit — `backend/app/ml/`

### `app/ml/features.py`
1. **Why created:** the single piece of code that must run *identically* at training time and serving time — the textbook fix for training/serving skew, and the most important architectural decision in the ML portion of this project.
2. **Problem solved:** if training-time encoding and serving-time encoding were two separate, hand-synced implementations, any drift between them (e.g. a different category ordering) would silently produce wrong predictions with no error raised — one of the most common real-world ML bugs.
3. **What breaks if removed:** `train.py` and `services/prediction.py` both fail to import; nothing in the ML pipeline functions.
4. **Connections:** `FeaturePipeline.fit_transform()` called once, offline, by `train.py`; `FeaturePipeline.transform()` (using the *same fitted encoder*, persisted via `encoder.joblib`) called on every request by `services/prediction.py`.
5. **Inputs:** a raw pandas DataFrame with the nine feature columns.
6. **Outputs:** an encoded DataFrame (categoricals → ordinal codes, numerics passed through).
7. **Dependencies:** `pandas`, `sklearn.preprocessing.OrdinalEncoder`, `joblib`.
8. **Depended on by:** `app/ml/train.py`, `app/services/prediction.py`.
9. **Design pattern:** the persisted-transformer pattern (fit once, serialize, reload identically) — standard scikit-learn production practice; also a light Builder (`fit_transform` vs `transform` mirrors scikit-learn's own Transformer API).
10. **Alternatives:** `pd.get_dummies()` one-hot encoding (would work for XGBoost too, but creates more columns and needs careful column-alignment between train/serve if categories are missing at inference time — ordinal encoding side-steps that since it's always exactly N columns); `sklearn.Pipeline` wrapping a `ColumnTransformer` (more "sklearn-idiomatic," functionally equivalent to the hand-rolled class here).
11. **Necessary or over-engineered:** necessary; appropriately minimal — resisting the urge to wrap this in a full `sklearn.Pipeline` for a 9-feature, single-table problem is the right call, not laziness.
12. **Recruiter read:** explicitly naming "training/serving skew" in the docstring and then visibly solving it is a strong, specific ML-engineering signal — most portfolio ML projects never even mention the concept.
13. **Senior engineer review:** would ask what happens when a *new* category appears at inference time that wasn't in training — answered correctly: `OrdinalEncoder(handle_unknown="use_encoded_value", unknown_value=-1)` maps it to `-1` rather than crashing. In practice this can't currently happen since the API layer's enums already restrict input to known categories — this handling is a defensive second layer, not the only line of defense.
14. **Improvement:** none structurally urgent; could add a unit test directly on `FeaturePipeline` (currently only exercised indirectly through `train.py` and the API integration tests).

### `app/ml/train.py`
1. **Why created:** the one script that turns a CSV into a deployed model — the entire "Phase 1" deliverable from the execution roadmap.
2. **Problem solved:** without it, there is no model — `services/prediction.py` would have nothing to load.
3. **What breaks if removed:** the model artifacts could never be regenerated; if `artifacts/model.json` were ever deleted or the dataset changed, the project would be unrecoverable without rewriting this file.
4. **Connections:** reads `Clean_Dataset.csv`; uses `features.py`'s `FeaturePipeline`; writes all three files under `app/ml/artifacts/` plus `docs/model_card.md`.
5. **Inputs:** `backend/data/raw/Air Line dataset/Clean_Dataset.csv` (300,153 rows).
6. **Outputs:** `model.json`, `encoder.joblib`, `model_metadata.json`, `docs/model_card.md`.
7. **Dependencies:** `pandas`, `numpy`, `scikit-learn`, `xgboost`, `app.ml.features`.
8. **Depended on by:** nothing imports this *as a module* — it's invoked directly (`python -m app.ml.train`), and its **outputs** are depended on by `services/prediction.py`. `backend/Dockerfile` also calls it during image build.
9. **Design pattern:** Script/Pipeline pattern — a linear, side-effecting `main()` rather than a class, appropriate for a one-shot offline job (no need for the OOP ceremony a long-running service would justify).
10. **Alternatives:** orchestrating this through Airflow/Prefect/a Makefile target (overkill for a single-model, single-dataset project); MLflow tracking (mentioned in V2_BLUEPRINT.md as a "nice to have," not yet implemented — currently metrics just go to a JSON file and a markdown report, which is sufficient for one model version but wouldn't scale to comparing many experiment runs).
11. **Necessary or over-engineered:** necessary; correctly un-engineered for a one-model project — adding MLflow now, before there's a second model to compare against, would be premature.
12. **Recruiter read:** the fact that running this script regenerates `docs/model_card.md` with *fresh, real numbers* every time (not a one-time hand-written report) is the kind of reproducibility detail that proves the metrics weren't fabricated after the fact.
13. **Senior engineer review:** would note the hardcoded `MODEL_VERSION = "v1.0.0"` never auto-increments — re-running training silently overwrites the same version string even if the underlying model changed, meaning `model_version` in API responses can't currently be trusted to mean "this exact model," only "some XGBoost model trained by this script."
14. **Improvement:** derive `MODEL_VERSION` from a content hash of the trained model or an incrementing counter persisted somewhere, so retraining actually produces a new, distinguishable version string — important the moment there's a second real training run.

### `app/ml/artifacts/` (model.json, encoder.joblib, model_metadata.json)
1. **Why created:** these are `train.py`'s **outputs** — the actual deployed model, not source code.
2. **Problem solved:** persisting the trained state so the FastAPI process doesn't need to retrain on every restart (which would take ~10+ seconds and require the full dataset to be present at runtime).
3. **What breaks if removed:** `PredictionService.__init__()` raises `FileNotFoundError` with an explicit message telling you to run `python -m app.ml.train` — a deliberately friendly failure, not a cryptic crash.
4. **Connections:** written by `train.py`, read by `services/prediction.py`.
5/6. **Inputs/Outputs:** binary/JSON artifacts, not human-edited.
7. **Dependencies:** produced by `xgboost`'s native serializer (`model.json`) and `joblib` (`encoder.joblib`).
8. **Depended on by:** `services/prediction.py`, `backend/Dockerfile` (which runs training during image build rather than shipping pre-trained artifacts — a deliberate choice so the container is self-contained and reproducible, at the cost of slower builds).
9. **Design pattern:** Model Registry, in its most minimal possible form (a folder, not a service) — appropriate for a single-model project; would need a real registry (MLflow Model Registry, S3-versioned artifacts) the moment there's more than one model version in play.
11. **Necessary or over-engineered:** necessary; correctly minimal.
14. **Improvement:** these binary artifacts are currently tracked the same way as source code (no `.gitignore` entry for them) — worth deciding deliberately whether to commit trained models to git (simple, but bloats repo history every retrain) or git-ignore them and regenerate via CI (cleaner, but means a fresh clone can't serve predictions until `train.py` is run once).

## 14. File-by-file audit — `backend/alembic/` (database migrations)

### `alembic.ini`
1. **Why created:** Alembic's required configuration file — log formatting, migration script location, and (originally) a hardcoded `sqlalchemy.url`.
2. **Problem solved:** without it, the `alembic` CLI has no idea where the migrations directory is or how to format its output.
3. **What breaks if removed:** every `alembic` command fails immediately with a "can't find config" error.
4. **Connections:** read by the `alembic` CLI; the `sqlalchemy.url` value it originally contained is **overridden at runtime** by `env.py` (`config.set_main_option("sqlalchemy.url", DATABASE_URL)`), so this file's own URL value is effectively dead/unused — a subtlety worth knowing if debugging "why didn't my DATABASE_URL change take effect."
9. **Design pattern:** standard Alembic project scaffolding (auto-generated by `alembic init`, lightly relied upon rather than hand-crafted).
14. **Improvement:** none needed; this is boilerplate working as intended.

### `alembic/env.py`
1. **Why created:** the actual Python logic Alembic runs on every migration command — this is the file that was **hand-edited** (not left as the auto-generated stub) to wire Alembic into this specific project's models and database URL.
2. **Problem solved:** the default `alembic init` output has `target_metadata = None`, meaning `--autogenerate` would detect zero schema changes, ever. This file imports `Base` and all three models and sets `target_metadata = Base.metadata`, making autogenerate actually work.
3. **What breaks if removed:** `alembic revision --autogenerate` would stop detecting any model changes (reverting to the "always empty migration" default); existing migrations would still apply fine, but the workflow that *creates* new migrations would be broken.
4. **Connections:** imports `app.core.database.Base`/`DATABASE_URL` and `app.models.*` — the bridge between "the app's actual models" and "what Alembic compares against to generate a migration."
5. **Inputs:** the current `DATABASE_URL` env var, the current state of `Base.metadata`.
6. **Outputs:** either applies migrations to the real DB (`upgrade`/`downgrade`) or generates a new migration file (`revision --autogenerate`).
7. **Dependencies:** `alembic`, `sqlalchemy`, `app.core.database`, `app.models`.
8. **Depended on by:** the `alembic` CLI exclusively; no application code imports this file.
9. **Design pattern:** standard Alembic environment script, customized at the two specific points (`target_metadata`, `sqlalchemy.url`) every real-world Alembic setup needs to customize.
10. **Alternatives:** skip Alembic entirely and use `Base.metadata.create_all()` on startup (works for a from-scratch dev DB, but provides no upgrade path for an existing production database with real user data — exactly the capability migrations exist to provide).
11. **Necessary or over-engineered:** necessary the moment there's a real deployed database with real data that needs to evolve without being dropped and recreated.
12. **Recruiter read:** "uses Alembic migrations" vs. "calls `create_all()`" is a real signal of having operated a database past the prototype stage.
13. **Senior engineer review:** would approve the `sys.path.insert` hack at the top (needed so `env.py`, which Alembic runs as a standalone script, can find the `app` package) — slightly unusual-looking but the standard workaround for this exact problem.
14. **Improvement:** none pressing.

### `alembic/versions/65c4edf4ebfd_create_users_watchlists_saved_searches.py`
1. **Why created:** the actual, single migration applied so far — auto-generated by `alembic revision --autogenerate`, creating the `users`, `watchlists`, `saved_searches` tables and the `ix_users_email` index in one migration.
2. **Problem solved:** gives the schema a version-controlled, reproducible creation history instead of "just run `create_all()` and hope."
3. **What breaks if removed:** `alembic upgrade head` would have nothing to apply — a fresh `app.db` would have zero tables, and every API route touching the database would fail with "no such table."
4. **Connections:** the only migration file that currently exists; every future schema change will be a *new* file building on top of this one's `down_revision` chain.
5. **Inputs:** none (it's a static generated file).
6. **Outputs:** the actual `CREATE TABLE` statements (via SQLAlchemy's DDL layer) when `upgrade()` runs.
7. **Dependencies:** `alembic`, `sqlalchemy`.
8. **Depended on by:** the live `app.db` schema; any future migration's `down_revision` chain starts here.
9. **Design pattern:** versioned schema migration (the standard Rails-migrations-inspired pattern Alembic implements for SQLAlchemy).
11. **Necessary or over-engineered:** necessary; auto-generated content shouldn't be hand-tuned beyond what Alembic produced (and wasn't, here).
14. **Improvement:** none — this file should essentially never be hand-edited again; future schema changes get *new* migration files.

### `alembic/script.py.mako`, `alembic/README`
Boilerplate: the Mako template Alembic uses to scaffold *future* migration files, and Alembic's own auto-generated README. Neither is hand-written project content; both are necessary only in the sense that deleting them would make `alembic revision` (no `--autogenerate`) fail to find a template. Zero recruiter/senior-engineer signal either way — pure tooling scaffolding.

## 22. Rollup — missing files (gaps vs. what's been designed/promised)

- No `routes`, `price_history`, `anomalies`, `api_usage`, `audit_logs` tables (designed in V2_BLUEPRINT.md Part 5, not yet implemented) — blocks Phase 5 entirely (fare calendar, real anomaly detection, real market analytics, admin dashboard).
- No scheduled job of any kind (no GitHub Actions workflow exists anywhere in this repo) — the entire "near-real-time price tracking" story is currently undeliverable, since there's no mechanism to populate `price_history` even if the table existed.
- No CI pipeline (`.github/workflows/` doesn't exist) — tests pass locally but nothing enforces that on a PR.
- No frontend tests (Jest/RTL scaffolding present via `react-scripts`, zero actual test files in `frontend/src/`).
- No email-sending capability — blocks email verification, password reset, and price-drop alert delivery, all three designed, none implemented.
- No `docs/decisions/` ADR folder despite V2_BLUEPRINT.md's roadmap explicitly calling for one starting at "Phase 0."
- No `.env.example` for the backend (frontend has a real `.env`; backend's `JWT_SECRET_KEY`/`DATABASE_URL` env-var contract is undocumented anywhere a new contributor would find it).
- No direct unit tests for `PredictionService` or `FeaturePipeline` in isolation — only exercised through HTTP integration tests.

## 23. Technical debt

1. **Two unmerged near-duplicate tables/schemas/routers** (`Watchlist` vs `SavedSearch`) — flagged independently in Sections 8, 9, and 10; the longer this persists, the more duplicated logic accumulates on top of it (e.g. if/when the alert-checking job is built, it'll need watchlist-specific logic that a merged table would have made shared).
2. **Manual enum duplication across the Python/JS language boundary** (`backend/app/schemas/predict.py` vs `frontend/src/api/client.js::FLIGHT_OPTIONS`) — silent-drift risk with zero compiler/test enforcement today.
3. **`requirements.txt` mixes two unrelated tech stacks** (Section 16) — makes `pip install` slower and the dependency graph misleading.
4. **`MODEL_VERSION` is a hardcoded string, not derived from model content** (Section 12) — retraining doesn't produce a distinguishable version.
5. **`backend/Dockerfile` retrains the model on every image build** rather than shipping pre-trained artifacts or using a multi-stage build.
6. **No frontend tests at all**, despite a real, tested backend — asymmetric test coverage across the stack.

## 24. Architectural weaknesses

1. **No refresh tokens / no token revocation** — a leaked JWT is valid for a full 24 hours with no way to invalidate it server-side (no token blocklist, no shorter-lived access token + refresh token pair).
2. **In-memory rate limiting** (Section 7, `core/rate_limit.py`) — breaks the moment there's more than one backend process/instance, since each process has its own counter.
3. **No email verification** — `signup` accepts any syntactically valid email with no proof of ownership.
4. **The OOD guard only checks two numeric features**, not unusual categorical *combinations* (Section 11) — a route/airline pair absent from training data is currently treated as equally trustworthy as a common one.
5. **Generation 1 and Generation 2 coexist with no clear boundary documented anywhere a new reader would find quickly** (fixed by this very document, but not yet reflected in the README itself).

## 25. Security risks

1. **`SECRET_KEY` has an insecure hardcoded fallback** (`"dev-only-insecure-secret-change-me"`) that would silently work if `JWT_SECRET_KEY` were forgotten in a real deployment — no fail-fast check exists.
2. **No HTTPS enforcement at the app layer** (expected to be handled by Vercel/Render's TLS termination once deployed — currently moot since nothing is deployed, but worth tracking as a deployment-time requirement, not an afterthought).
3. **No account lockout after repeated failed logins** beyond the generic 10/minute rate limit — a determined attacker gets 10 password guesses per minute indefinitely, never fully locked out.
4. **No CSRF concern currently** (JWT-in-header, not cookie-based, so CSRF is structurally not applicable — correctly so, just worth confirming explicitly rather than by accident).
5. **`backend/app.db` (SQLite) currently has no encryption at rest** — acceptable for a local dev database with synthetic test users, would need revisiting if real user data were ever stored in it.

## 26. Scalability bottlenecks

1. **SQLite + single-process in-memory rate limiter** — both fundamentally single-instance assumptions; both have a documented, cheap fix already designed in V2_BLUEPRINT.md (Postgres via one env var; Redis-backed slowapi storage) but neither fix has been executed yet.
2. **Model held entirely in one process's memory, loaded once at startup** — fine for one instance; horizontally scaling to N instances means N redundant copies of the same ~few-MB model in memory, which is acceptable at this model's size but is the kind of assumption that breaks down with a much larger model.
3. **No caching layer** (Upstash Redis designed in the blueprint, not yet wired in) — every `/predict` call re-runs inference even for identical, frequently-repeated queries (e.g. the same popular route searched by many users).
4. **No pagination on `GET /watchlists` or `GET /saved-searches`** — fine at a 20/30-item per-user cap today, would need pagination if those caps were ever raised.

## 27. Refactoring opportunities (ranked by effort vs. payoff)

1. **Lowest effort, real payoff:** wire `Profile.js`/`Settings.js`/`Security.js`/`Notifications.js` to the already-built, already-tested `/auth/me`, `/watchlists`, `/saved-searches` endpoints — no new backend code needed, purely frontend wiring (same pattern already proven in `PricePrediction.js`).
2. **Low effort:** delete or clearly relabel `backend/src/`, `backend/infrastructure/`, root `SYSTEM_DESIGN.md` per Section 19 — pure documentation/cleanup, no code risk.
3. **Medium effort:** merge `Watchlist`/`SavedSearch` into one table+schema+router pair with a `kind` discriminator — one new migration, then delete one of the two near-duplicate router files.
4. **Medium effort:** generate `frontend/src/api/client.js`'s `FLIGHT_OPTIONS` from the backend's live OpenAPI schema instead of hand-maintaining a duplicate.
5. **Higher effort, highest payoff:** build the `price_history` table + a GitHub Actions scheduled job to populate it — this single piece of infrastructure unblocks Fare Calendar, real Anomaly Detection, real Market Analytics, and the Watchlist alert-checking job all at once, since all four currently-mocked features share this one missing dependency.

## 28. Teaching this to a new engineer, Day 1

If you're joining this project cold, here's the mental model:

**Ignore `backend/src/`, `backend/infrastructure/`, and `SYSTEM_DESIGN.md` for now.** They describe a different, much bigger system (AWS Kinesis/Glue/SageMaker at 10M-searches/day scale) that was never built. They're not wrong, they're just not *this* system. Treat them as a "if we ever needed to scale way up, here's a sketch" reference, not onboarding material.

**The real system is `backend/app/` + one page of `frontend/src/`.** Start by running it: `cd backend && venv\Scripts\python -m uvicorn app.main:app --reload` and `cd frontend && npm start`, then open `localhost:3000`, click "Price Prediction," and fill out the form. That round-trip — React form → axios POST → FastAPI route → Pydantic validation → `PredictionService` → XGBoost model → JSON response → React renders a price — is the one fully real, fully working slice of this whole project. Everything else, including ten of the twelve sidebar tabs, is either mock data (V1 leftover) or built-but-unwired backend capability (auth, watchlists, saved searches — real, tested, just not called from the UI yet).

**The model is not magic.** `app/ml/train.py` is a 150-line script: load a CSV, encode nine columns, fit an XGBoost regressor, measure error on data it didn't train on, save the result. Run it yourself (`python -m app.ml.train`) and watch `docs/model_card.md` regenerate with real numbers. If that file's numbers ever look suspicious, this is the only script that could have produced them — there's no hidden data pipeline.

**Auth is real but disconnected from the UI.** You can `curl -X POST localhost:8000/api/v1/auth/signup` right now and get back a real JWT, real bcrypt-hashed password in `app.db`. No `Login` button exists in the React app yet — that's the very next piece of work, and it's wiring, not invention, since the backend side is done and tested.

**Tests are the source of truth for "does this actually work."** `cd backend && pytest tests/ -v` — 23 tests, and reading them top to bottom (auth → predict → watchlists → saved searches → rate limiting) is a faster way to understand the system's actual behavior than reading the route code directly, because the tests show both the happy path *and* what's deliberately disallowed (cross-user deletes, invalid enums, rate limit trips).

**The biggest open risk if you're about to demo this live:** don't click into Demand Forecasting, Market Analytics, Anomaly Detection, Data Sources, or System Health unless you explicitly frame them as "design mockups, not yet wired to real data" — they will move/animate convincingly and an unprepared viewer will assume they're real, which then casts doubt on the one tab that *is* real the moment they figure out the others aren't.
